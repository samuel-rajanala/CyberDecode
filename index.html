Reference

Format specs, use cases, and real-world examples.

B64
Base64
Alphabet: A–Z a–z 0–9 + /
Padding: = to multiple of 4
Output: ~133% of input
Use: JWT tokens, data URIs, MIME
B64U
Base64 URL
Replaces +→- and /→_
No padding characters
Safe in URLs and filenames
Use: OAuth, API keys, cookies
B32
Base32
Alphabet: A–Z 2–7
Case-insensitive, no ambiguous chars
Output: ~160% of input
Use: TOTP (2FA), DNS labels
HEX
Hexadecimal
Each byte → 2 hex digits 00–FF
Upper or lowercase output
Output: 200% of input
Use: Binary dumps, hashes, MAC
URL
URL Encoding
Unsafe chars → %HH
Space → %20 (RFC 3986)
Standard percent-encoding
Use: Query strings, path params
FORM
Form Encoding
Space → + instead of %20
application/x-www-form-urlencoded
Used by HTML form POST
Use: Login payloads, forms
Real-world examples
JWT header · Base64 URL
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9
→ {"alg":"HS256","typ":"JWT"}
TOTP secret · Base32
JBSWY3DPEHPK3PXP
Used by Google Authenticator / Authy
URL query string · Form encoded
q=hello+world&lang=en%2FUS
→ q="hello world", lang="en/US"
SHA-256 hash · Hex
2cf24dba5fb0a30e26e83b2ac5b9e29e1b161e5c
→ sha256("hello") (truncated)
Detection heuristics
Base64
Chars [A-Za-z0-9+/=], length % 4 = 0, padding present
Base64 URL
Chars [A-Za-z0-9-_], no + / = padding
Base32
Chars [A-Z2-7=], length % 8 = 0
Hex
Chars [0-9A-Fa-f: ], even length
URL / Form
Presence of %XX sequences; + for spaces in form
Encod3r — encoding is not encryption — no security is provided
